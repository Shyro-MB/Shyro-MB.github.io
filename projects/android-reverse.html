1. Acquisition

Récupérer l’APK depuis l’appareil ou une source fiable, puis vérifier l’intégrité et le contexte (version, split APK, signature, etc.).

Méthodes

  • Via ADB — depuis un PC connecté
  • Via Termux — directement sur l'appareil
adb shell pm list packages | grep "nom"
adb shell pm path com.example.app
adb pull /data/app/com.example.app-xxx/base.apk

Astuce : pour les applications splitées, pm path peut retourner plusieurs APKs.

2. Vue d’ensemble (JADX)

Étape rapide de cartographie : structure, bibliothèques natives, signaux d’obfuscation/protection, composants Android, points d’entrée, etc.

  • Présence d’un stub / code minimal
  • Indices natifs via System.loadLibrary()
  • Noms de classes/paquets atypiques
  • Organisation globale et surface d’attaque

Exemple : indice de charge native

public class MainActivity extends Activity {
  static { System.loadLibrary("native_core"); }
  @Override protected void onCreate(Bundle b) {
    super.onCreate(b);
    init();
  }
  private native void init();
}

Note : JADX décompile (dex → Java). Pour le bas niveau et les modifications : smali.

3. Extraction & désassemblage (APKTool)

Extraction des ressources + manifest, et désassemblage du bytecode en smali (lecture fine du flux d’exécution). 

apktool d app.apk -o app_sources/
app_sources/
├── smali/
├── smali_classes2/
├── res/
├── AndroidManifest.xml
└── apktool.yml

Précision : APKTool désassemble (dex → smali). La vue Java vient de jadx.

4. Analyse statique (Smali + JADX)

Analyse détaillée des flux d’exécution et des contrôles applicatifs. Les deux vues sont complémentaires : JADX pour la lecture, smali pour la précision bas niveau.

Exemple : analyse d’un mécanisme de validation applicatif

Vue JADX (Java)

// Exemple simplifié : contrôle applicatif (feature flag / validation interne)
if (validator.isAllowed(context)) {
  enableFeature();
} else {
  showRestrictedState();
}

Vue Smali (flux logique correspondant)

invoke-virtual {v0, p0}, Lcom/example/Validator;->isAllowed(Ljava/lang/Object;)Z
move-result v1
if-eqz v1, :cond_restricted
invoke-direct {p0}, Lcom/example/MainActivity;->enableFeature()V
goto :end
:cond_restricted
invoke-direct {p0}, Lcom/example/MainActivity;->showRestrictedState()V
:end

5. Identification des protections

Fingerprinting des protections et des architectures (obfuscation, loaders, Unity, etc.) pour orienter la suite de l’analyse.

Exemples de signaux

  • Charge native : libs spécifiques, init natif, stubs minimalistes
  • Obfuscation : classes/strings atypiques, packages artificiels
  • Unity : libil2cpp.so, structure assets

6. Instrumentation dynamique (Frida)

L’instrumentation dynamique sert à observer le comportement runtime lorsque l’analyse statique ne suffit pas (chargements dynamiques, logique déplacée, etc.).

Installation (exemple)

pip install frida-tools

But : observation & compréhension (traces, hooks, timings), dans un cadre d’audit et de recherche.

7. Observabilité & modification contrôlée

Dans un contexte d’audit, il est souvent utile d’ajouter de l’observabilité (logging/trace) pour confirmer un flux d’exécution, mesurer des conditions et documenter le comportement.

Exemple : ajout de trace (confirmation de branche)

Avant

invoke-virtual {v0, p0}, Lcom/example/Validator;->isAllowed(Ljava/lang/Object;)Z
move-result v1
if-eqz v1, :cond_restricted

Après (trace de diagnostic)

invoke-virtual {v0, p0}, Lcom/example/Validator;->isAllowed(Ljava/lang/Object;)Z
move-result v1

# Observabilité : log de diagnostic (ex. "allowed=true/false")
const-string v2, "SHIRO-AUDIT"
if-eqz v1, :cond_log_false
const-string v3, "allowed=true"
goto :cond_log
:cond_log_false
const-string v3, "allowed=false"
:cond_log
invoke-static {v2, v3}, Landroid/util/Log;->d(Ljava/lang/String;Ljava/lang/String;)I

if-eqz v1, :cond_restricted

Pourquoi : confirmer un comportement, produire une preuve d’exécution, améliorer la documentation d’audit.

8. Rebuild & signature

Reconstruction contrôlée après modifications (ressources + smali → APK), puis signature. 

apktool b app_sources/ -o modified-unsigned.apk
apksigner sign --ks my-key.keystore modified-unsigned.apk

Note : apktool b assemble le smali en dex puis reconstruit l’APK.

9. Tests

Validation sur appareil / environnement de test.

adb install modified.apk

10. Documentation & outils

Production d’un write-up clair et d’outils d’analyse pour automatiser la collecte d’indicateurs.

Outils développés

manifest-analyzer.py

Analyse AndroidManifest.xml.

python manifest-analyzer.py app.apk

Recherche de patterns smali.

python smali-pattern-matcher.py ./smali/

dex-dumper.js

Script Frida (analyse mémoire / collecte d’artefacts).
