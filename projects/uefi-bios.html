1. Firmware Acquisition

Première étape : obtenir une image exploitable du firmware, soit par dump local, soit via un firmware public fourni par le constructeur.

Méthodes

  • Dump local — acquisition via flashrom
  • Firmware public — fichier .bin ou .cap récupéré depuis le constructeur
# Lecture du firmware avec flashrom
sudo flashrom -p internal -r bios.bin

Approche utilisée : analyse en lecture seule d’un firmware réel, sans modification ni réécriture. 

firmware.bin
├── taille : 16 MB
├── source : dump local ou constructeur
└── format : .bin / .cap

2. Firmware Parsing — UEFITool

Une fois l’image du firmware obtenue, l’étape suivante consiste à ouvrir sa structure interne : volumes, modules, drivers et zones NVRAM.

Ce qu’on analyse

  • Firmware Volumes (FV)
  • Modules PEI
  • Modules DXE
  • Drivers SMM
  • Volume NVRAM
Firmware Volume (FV)
├── PEI volume
│   ├── PEI Core
│   └── PEIM modules
├── DXE volume
│   ├── DXE Core
│   ├── DXE drivers
│   └── SMM drivers
└── NVRAM volume

Point clé : cette étape confirme qu’on travaille sur une image réelle, avec des volumes et modules effectivement présents dans le firmware.

3. Structure Analysis — PEI / DXE / SMM

Le firmware suit une logique d’exécution en plusieurs phases. L’objectif ici n’est pas de refaire la spécification UEFI, mais de relier les concepts aux composants observés dans l’image.

Phase Rôle
PEI Initialisation précoce du système avant l’environnement principal.
DXE Chargement des drivers et services UEFI principaux.
SMM Contexte très privilégié pour certaines opérations sensibles.

Dans la pratique, l’arborescence UEFITool permet de visualiser ces différentes couches et d’identifier rapidement les modules les plus intéressants à extraire.

4. NVRAM Analysis — BootOrder / BootXXXX

Les variables NVRAM décrivent l’ordre de démarrage et les différentes entrées de boot. Elles constituent un point d’observation important dans la chaîne pré-OS.

Sous Windows

bcdedit /enum firmware

Exemple concret

BootOrder: 0001, 0002
Boot0001: Windows Boot Manager
Boot0002: USB Drive

Lecture : BootOrder définit la priorité de démarrage, tandis que les variables BootXXXX décrivent chaque entrée disponible.

5. Secure Boot — Chaîne de confiance

L’UEFI ne se limite pas à lancer un chargeur. Il applique aussi une logique de validation basée sur une hiérarchie de clés et de bases de signatures. 

PK (Platform Key) → KEK (Key Exchange Key) → DB (Signature DB) → DBX (Revoked DB)

Vérification sous Windows

Confirm-SecureBootUEFI
Get-SecureBootUEFI
  • PK — racine de confiance plateforme
  • KEK — clés pouvant mettre à jour les bases
  • DB — signatures autorisées
  • DBX — signatures révoquées

Point d’analyse : la sécurité ne dépend pas seulement d’un état activé ou désactivé, mais de la qualité des clés et de la politique de validation en place.

6. Binary Analysis — Driver UEFI dans Ghidra

Après parsing du firmware, certains drivers DXE peuvent être extraits puis chargés dans un analyseur binaire afin d’observer leur logique d’initialisation et les protocoles utilisés.

Extraction du driver

  • Clic droit sur un module dans UEFITool
  • Extract body
  • Sortie : fichier .efi

Ouverture dans Ghidra

  • Loader adapté au binaire EFI
  • Architecture 32 ou 64 bits selon le module
  • Recherche de fonctions, GUIDs et protocoles EFI
EFI_STATUS EFIAPI GraphicsOutputDriver(
    EFI_HANDLE ImageHandle,
    EFI_SYSTEM_TABLE *SystemTable)
{
    EFI_STATUS Status;
    Status = gBS->InstallMultipleProtocolInterfaces(
        &ImageHandle,
        &gEfiGraphicsOutputProtocolGuid,
        &mGraphicsOutput,
        NULL);
    return Status;
}

7. Attack Surface Mapping

Le firmware représente une surface d’exécution persistante, indépendante du système d’exploitation. Cette couche mérite donc une lecture sécurité à part entière.

Composant Risque principal
DXE Drivers Bugs logiques, parsing vulnérable, erreurs de gestion mémoire.
NVRAM Modification de BootOrder ou de variables sensibles.
SMM Surface hautement privilégiée, critique en cas d’erreur.
Secure Boot Configuration faible, clés mal gérées ou politique de validation insuffisante.

Cas concret : si l’ordre de boot peut être modifié depuis un environnement insuffisamment protégé, il devient possible de rediriger le démarrage vers un autre chargeur.

8. Case Study — Firmware MSI réel

Cette analyse a été réalisée à partir d’un firmware MSI réel, afin de confronter les concepts UEFI à une implémentation concrète. 

Firmware analysé : MSI (BIOS officiel)

Observations :
- Présence de plusieurs DXE drivers identifiables
- Modules SMM visibles dans l’arborescence
- Variables NVRAM observables depuis Windows
- Secure Boot présent selon la configuration système

Conclusion :
Le firmware suit une architecture UEFI classique avec séparation nette
des composants d’initialisation, des drivers principaux et de la logique
de confiance appliquée au démarrage.

Conclusion

Le firmware n’est pas simplement une étape de démarrage. Il constitue la première surface d’exécution du système, initialise des composants critiques, applique une logique de confiance et prépare la transition vers l’OS.

Cette analyse met en évidence une couche rarement étudiée dans les projets classiques, mais essentielle pour comprendre la sécurité globale d’une machine.
